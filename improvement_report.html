1. このレポートについて

アライメント計測・Shot補正可視化ダッシュボードの開発は、要件定義 → 画面モックアップ → PBIPプロジェクト実装（P1〜P7のビジュアル配置）まで進みました。 本レポートは、この時点のリポジトリを見直して「次にやると効果が大きい改善」を 優先度（高・中・低）と工数目安（小: 半日以内 / 中: 1〜2日 / 大: 3日以上）つきで整理したものです。

※ 課題番号（I-xxx）は issues.md の管理番号、V-xx は requirements.md の可視化項目番号に対応します。 提案番号は K-01〜K-16 です。

4. 優先度: 高

K-01 Windows再検証とDeneb実装枠の置き換え完了 高

工数目安: 中（Deneb置き換えを含む）

背景
開けない不具合はPR #6で修正しましたが、修正後にWindowsのPower BI Desktopで開けることの確認がまだです。また、ベクトル場・共分散楕円・補正前Shot配列・箱ひげの13枠は「Deneb実装枠」のままで、グラフとして動いていません。
提案
①最新mainをWindowsで開き、全ページの表示を確認 → ②枠に書かれた手順どおりDenebビジュアルへ置き換え（P3の計測値ベクトルマップから着手推奨）→ ③置き換え後にDesktopから保存し、差分をPRでレビュー。
期待効果
ダッシュボードの中核であるWaferマップ系が実際に動く状態になり、以降の調整（倍率・配色）が実データ確認と並行して進められます。

K-02 装置エンジニア確認の完了（I-001b / I-002 / I-009） 高

工数目安: 小（確認依頼の実施。回答待ちが主）

背景
Vertica接続情報（I-001b）が無いと本番切替ができません。単位・判定0/1の意味（I-002）が未確定のままだと、カードやツールチップの単位表記（nm/ppm/µrad）が想定値のままになります。補正前逆算の符号規約（I-009)が未確定だと、V-18/V-26で補正効果が逆向きに見えるリスクがあります。
提案
issues.mdの該当3件を「確認依頼票」として1枚にまとめ、装置エンジニアへ送付。回答が届いた項目から sys_emas.md・メジャーの単位表記・deneb定義に反映します。
期待効果
実装のブロッカーが解消し、誤読リスク（単位違い・符号逆）を早期に潰せます。

K-03 期間スライサーの既定値（直近7日）と負荷ガード 高

工数目安: 小

背景
要件では「必須フィルター未選択のまま全期間クエリをVerticaへ飛ばさない」としていますが、現状のスライサーには既定値がなく、開いた直後に広い範囲のクエリが発生し得ます。
提案
①期間スライサーを「相対日付: 直近7日」を既定に設定 ②各ページに「Wafer数」「測定点数」の件数ガードカードを置き、絞り込み不足を数字で気づけるようにする ③マトリックス系にTop Nフィルターを設定。
期待効果
DB負荷と初期表示時間を抑え、ページロード3秒目標に近づけます。DirectQuery本番切替後の事故（全期間スキャン）を防ぎます。

K-04 スライサー同期 高

工数目安: 小（Desktop操作のみ）

背景
7ページ × 8本のスライサーは現状ページごとに独立しています。P1で選んだ期間や装置がP2以降に引き継がれず、ページを移るたびに選び直しが必要です。
提案
Desktopの「スライサーの同期」で、期間・DEVICE_TYPE・OPERATION・EQPID等をページ間同期します（Canon系とNikon系でファクト列が異なるため、同期グループはCanon系/Nikon系/共通の3グループに分けるのが安全です）。
期待効果
「俯瞰→詳細」の分析動線が1回の絞り込みで通り、操作ミス（ページごとに条件が違う状態での誤読）を防げます。

K-05 ドリルスルー設定（P2→P3、P5→P6） 高

工数目安: 小

背景
要件の分析動線は「俯瞰ページで気になる装置・Lotを右クリック → 詳細ページへドリルスルー」ですが、ドリルスルーフィールドが未設定です。
提案
P3・P6にドリルスルーフィールド（EQPID・LOT_ID・OPERATION）を設定し、P2・P5から遷移できるようにします。あわせて「戻る」ボタンを配置します。
期待効果
モックアップで説明していた操作がそのまま使えるようになり、利用者の教育コストが下がります。

5. 優先度: 中

K-06 PROCESS / PRODUCT / DEVICE_TYPE のディメンション化 中

工数目安: 中

背景
現状これらのスライサーはCanon/Nikonのファクト列を直接使っており、ページによって参照テーブルが異なります。装置（EQPID）だけは統合ディメンションがありますが、他は未対応で、スライサー同期（K-04）とも相性が悪い構成です。
提案
装置テーブルと同じ方式（両ビューからDISTINCT取得のPower Queryテーブル）で工程・製品・デバイス種別のディメンションを作り、スライサーをそちらへ差し替えます。DB側にマスタ表があればそれを参照します（I-007）。
期待効果
1本のスライサーでCanon/Nikon両方を絞れるようになり、同期設定も単純になります。

K-07 日付テーブルの範囲を動的化 中

工数目安: 小

背景
日付テーブルは2026/1/1〜2027/12/31の固定生成です。2028年以降のデータが来るとリレーションから漏れ、グラフから静かにデータが消える恐れがあります。
提案
Mクエリを「今日を基準に過去N年〜未来1年」を生成する式へ変更します（例: Date.AddYears(Date.From(DateTime.LocalNow()), -2) 開始）。
期待効果
保守作業ゼロで日付切れを防止できます。

K-08 重複データ平均の実現方式を確定（I-006） 中

工数目安: 中（方式判断+実装）

背景
要件では「同一WaferId・計測日時・Shot座標・Mark座標の重複は平均してから集計」としていますが、現行メジャーは重複をそのまま平均しており、再測定があった場合に統計値がわずかにズレます。
提案
優先順位: ①ビュー側で重複排除済みビューを用意してもらう（DirectQueryと最も相性が良い）→ ②難しければSUMMARIZEで2段階集計するDAXメジャーへ変更（負荷は実測必須）。どちらにするかをI-003のデータ量確認とセットで決めます。
期待効果
RMSEやばらつき系指標（共分散楕円含む）の正確性が要件どおりになります。

K-09 判定0率のY成分追加とV-04の拡張 中

工数目安: 小

背景
判定関連はX成分（RAW_DATA_STSX）のみ実装しており、Y成分（STSY）のカード・比率グラフがありません。
提案
「判定0率Y」メジャーを追加し、P1のカードをX/Y並記へ、V-04の積み上げバーもX/Y切替（またはX/Y並置）にします。0/1の意味確定（I-002）後にラベルを正式名称へ変えます。
期待効果
X/Y片側だけ判定不良が増えるケース（照明条件・Mark依存など）を見逃さなくなります。

K-10 PBIP定義ファイル検証のCI化 中

工数目安: 中

背景
「Desktopで開けない」不具合の原因は、JSONの必須項目欠落とスキーマURL誤りでした。これはMicrosoft公式スキーマとの突き合わせで機械的に検出できることが今回実証済みですが、現在は手元での手動実行です。
提案
GitHub Actions（Privateリポジトリ）でPRごとに ①全JSONの構文チェック ②公式スキーマ検証 ③ファイル参照の実在チェック（今回作成したスクリプト相当）を自動実行し、失敗したらマージ不可にします。
期待効果
「マージ後にWindowsで開いて初めて壊れていると分かる」手戻りを、PR段階で防げます。

K-11 ツールチップページ・明細ドリルスルーページ 中

工数目安: 中

背景
要件の補足ページ（マウスオーバーでLot/Wafer基本情報、Top N上限つき生データ明細）が未実装です。
提案
ツールチップ用の小型ページ（Lot/Wafer/装置/件数のカード構成）と、明細テーブルページ（Top N 1,000行上限+CSVエクスポート抑止設定）を追加します。
期待効果
グラフ上で気になった点の一次情報（どのLot/Waferか）へ即座に到達でき、調査時間を短縮します。

K-12 比較軸（号機/製品/工程）切替の仕組み 中

工数目安: 中

背景
トレンドの凡例は現状固定です。要件の「号機別・製品別の比較」を1つのグラフで切り替えられると、ビジュアル数を増やさずに分析の幅が広がります（1ページ8個目安の対策にもなります）。
提案
フィールドパラメーター（Desktopの標準機能）で凡例を EQPID / PRODUCT / PROCESS から選択式にし、P1/P4のトレンドへ適用します。
期待効果
ページを増やさずカテゴリ比較の要件を満たせ、P1/P4のビジュアル過密も緩和できます。

6. 優先度: 低

K-13 サンプルCSVの軽量化 低

工数目安: 小

背景
ダミーCSVが約24MB（CANON 約19MB + NIKON 約4MB）あり、クローンやPR差分表示がやや重くなっています。
提案
生成スクリプトにLot数・Wafer数を絞る引数（例: --lots 3）を追加し、リポジトリ同梱版は数MBに縮小。フルサイズは必要時に各自生成する運用にします。
期待効果
リポジトリ操作が軽くなり、生成条件の再現もしやすくなります。

K-14 テーマをレポートに組み込み 低

工数目安: 小

背景
themes/theme.json は「Desktopで手動適用」の手順になっており、適用忘れで配色が既定のままになる可能性があります。
提案
Desktopでテーマ適用後に保存し、レポート定義（RegisteredResources）としてリポジトリへ取り込みます。以後はテーマ変更もPR差分でレビューできます。
期待効果
誰が開いても同じ配色・フォントになり、スクリーンショット比較も安定します。

K-15 Deneb表示速度の実測と行数上限の運用値決定 低

工数目安: 小（K-01完了後）

背景
要件で「カスタムビジュアルは本番投入前に表示速度を実測」とされていますが未実施です。DirectQueryのビジュアル行数上限（既定30,000行）に対し、1Lotの測定点は最大約12,000点です。
提案
パフォーマンスアナライザーで、Wafer1枚/1Lot/複数Lotの3条件×各Denebビジュアルの描画時間を計測し、「何枚まで重ねて良いか」の運用値をdesign.mdに記録します。
期待効果
ページロード3秒目標の達成可否を数字で判断でき、Aggregations等の追加対策の要否を早期に決められます。

K-16 schedule.json（ガントチャート用）の作成 低

工数目安: 小

背景
標準ファイル構成のうち schedule.json が未作成です。残作業（K-01〜K-15とエンジニア確認の待ち時間）が見える化されていません。
提案
指定テンプレートに従い、本レポートの提案を依存関係つきのスケジュールへ落とし込みます（例: K-02の回答 → 単位確定 → Deneb調整、の依存）。
期待効果
「いつ何が終わるか」「何が回答待ちで止まっているか」を関係者と共有できます。

7. 推奨する進め方

  1. 今週: K-01（Windows再検証）と K-02（エンジニアへ確認依頼を送付）を並行で開始。開けることを確認したら K-03〜K-05（既定値・同期・ドリルスルー）はDesktop操作のみでまとめて実施できます
  2. 回答待ちの間: K-10（CI化）・K-07（日付動的化）・K-09（判定Y）・K-13（CSV軽量化）など、確認回答に依存しない改善を消化
  3. 回答が揃ったら: 単位・符号をモデルとDeneb定義へ反映 → K-08（重複平均）と K-15（速度実測）→ Vertica DirectQuery切替（design.md 2.3）
  4. 公開前: RLS要否判断（I-004）、K-11（明細ページのTop N上限）、データ更新スケジュールの業務時間外設定を最終確認

※ どの改善も「ブランチ → PR → 確認 → マージ」の手順で進めます。見た目が変わるものはPRにスクリーンショットを添付します。
