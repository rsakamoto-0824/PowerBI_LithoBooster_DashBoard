# LithoBooster アライメント計測・Shot補正 可視化ダッシュボード 要件仕様書

## 1. 目的・背景

半導体製造の露光工程において、LithoBoosterが収集したアライメント計測結果と
Shotごとの補正制御値を、Power BIダッシュボードでインタラクティブに可視化する。

- 装置エンジニア・プロセスエンジニアが、Lot / Wafer / Shot単位で
  アライメントの傾向と補正の効き方を素早く確認できるようにする
- 装置間（号機間）・製品間・工程間の差を比較できるようにする
- データはデータベースから**ダイレクトクエリ（DirectQuery）**で直接取得し、
  ファイルへのデータ取り込み（Import）を行わない

## 2. データソース

### 2.1 接続先

- データベース: **Vertica**
- スキーマ: `sys_emas`
- 対象ビュー（2本）:

| ビュー名 | 対象装置 | 内容 |
|---|---|---|
| `V_EQP_WAFER_LITHOBOOSTER_CANON` | Canon露光機 | AGA計測結果（Mark単位の生データ含む）、AGA線形成分、PSO統計値、Shot形状補正（SEPA / SAME） |
| `V_EQP_WAFER_LITHOBOOSTER_NIKON` | Nikon露光機 | Wafer線形成分（Offset / Scaling / Rotation）、Shot線形成分、補正点データ（CORRDATA）、Shot高次補正係数（SHOTFAC） |

- 列定義の正は [sys_emas.md](sys_emas.md) とする（各列の想定する意味・単位も同ファイルに記載。※要確認の項目は I-002 で管理）
- 接続情報（サーバー名・ポート・DB名）は未確定（→ issues.md I-001）

### 2.2 接続方式

- **DirectQueryを必須とする**（データの持ち出し・キャッシュ蓄積を避けるため）
- Power BI標準の**Verticaコネクタ**を使用する（DirectQuery対応）
- Power BI Service公開時はOn-premises Data Gatewayが必要。
  ゲートウェイマシンにVertica ODBCドライバーのインストールが必要（→ I-001）
- 認証情報はレポートファイルやリポジトリに含めない。接続時の資格情報は
  Power BI / ゲートウェイ側の資格情報管理に任せる

### 2.3 データ粒度

- Canonビュー: 1行 = 1 Wafer × 1 Shot × 1 Mark（MARKINFO_MARKNO単位）
- Nikonビュー: 1行 = 1 Wafer × 1 補正点（CORRDATA_ID単位）
- 共通キー候補: `LOT_ID`, `WAFER_ID`, `WAFER_NO`, `MEASUREMENT_TIME`, `EQPID`
- 共通属性: `TEC_KIND`, `PROCESS`, `OPERATION`, `PRODUCT`, `DEVICE_TYPE`, `CTRL_JOB`

### 2.4 前提条件（Overlay計測 共通ルール準拠）

- 300mm Waferを前提とし、Wafer中心を座標原点 (0, 0) とする
- Wafer有効半径は150mmとし、半径150mm以上の点は表示対象外とする
- 座標単位はmm、計測値単位はnmを基本とする
  （ビュー各列の実際の単位は要確認 → I-002）
- 計測日時はJSTとする
- 1Lotは最大25 Wafer、WaferあたりShot数は90〜120程度、
  1Shotあたり計測Markは通常4点（端Shotは少ない場合あり）を想定する
- 通常ケースで1Lotあたり最大約12,000測定点を想定する

## 3. 想定ユーザーと利用シーン

| ユーザー | 利用シーン |
|---|---|
| 装置エンジニア | 号機ごとのアライメント傾向監視、異常号機の切り分け |
| プロセスエンジニア | 製品・工程ごとの補正値トレンド確認、条件変更前後の比較 |
| 工程管理者 | Lot単位の傾向把握、異常Lotの早期発見 |

## 4. 機能要件

### 4.1 共通フィルター（スライサー）

すべてのページで以下の条件で絞り込みできること。
**必須フィルター**は常に選択された状態で使う運用とし、未選択のまま
全期間・全装置のクエリがVerticaに飛ばないようにする。

**必須（常に絞り込む）:**

- 期間（`MEASUREMENT_TIME`、日付範囲。既定値: 直近7日）
- `DEVICE_TYPE`（デバイス種別）
- `EQPID`（装置・号機）
- `OPERATION`

**任意（必要に応じて絞り込む）:**

- `LOT_ID`
- `WAFER_ID` / `WAFER_NO`
- `PROCESS`（工程）
- `PRODUCT`（製品）

実装上の注意: Power BIのスライサーには「必須入力」を強制する標準機能が
ないため、①必須スライサーに既定値を設定する、②未選択時に件数が多く
なりすぎないようレポートフィルターで上限を設ける、の両方で担保する。

### 4.2 ページ構成（案）

CanonとNikonでスキーマが大きく異なるため、ページは装置メーカー別に分ける。
1ページのビジュアルは8個以下を目安とし、詳細はドリルスルーで表示する。

#### P1: サマリー / トレンド（Canon）

- AGA線形成分（`AGA_LINEAR_MX/MY/TX/TY/SX/SY`）のLot・Wafer時系列トレンド
- PSO統計値（`PSO_LINEAR_*`、3σ含む）の時系列トレンド
- 号機別・製品別の比較（折れ線 + 凡例、または小型マルチ）
- 管理基準値（しきい値）ラインの表示（基準値は要確認 → I-005）

#### P2: Waferマップ（Canon）

- Shot配置マップ上に計測残差（`RAW_DATA_X/Y`）をベクトル表示
- AGAサンプルShot（`IS_AGA_SAMPLE_SHOTS`）の識別表示
- Mark単位の値をShot単位に集計した色分けマップ（ヒートマップ）
- 表示するWaferはスライサーで1枚選択する方式とする

#### P3: Shot補正詳細（Canon）

- Shot形状補正の比較: `SEPA_SHOTFORM_*` / `SEPA_SHOTDATA_*` と
  `SAME_SHOTFORM_*` / `SAME_SHOTDATA_*`
- Shot番号（`SHOTINFO_SHOTNO`）別の補正量表示
- P1 / P2からのドリルスルー先とする

#### P4: サマリー / トレンド（Nikon）

- Wafer線形成分（`WAFER_OFSETX/Y`, `WAFER_SCALX/Y`, `WAFER_ROTX/Y`）の時系列トレンド
- Shot線形成分（`SHOT_OFSETX/Y`, `SHOT_SCALX/Y`, `SHOT_ROTX/Y`）の時系列トレンド
- 号機別・製品別の比較

#### P5: 補正マップ / 高次係数（Nikon）

- 補正点データ（`CORRDATA_POSX/POSY` 上に `CORRDATA_OFSETX/Y` 等）のベクトル・マップ表示
- Shot高次補正係数（`SHOTFAC*`）の係数別トレンド・比較表示

#### P6: Shotスタック分析

- 全ShotをShot座標系（Shot中心原点）に重ねて、Shot内の系統的な傾向を表示
- 補正前後の切り替え表示（→ 4.5 観点D・E）

#### 補足ページ

- ツールチップページ: マウスオーバーでLot / Wafer基本情報を表示
- ドリルスルーページ: 生データ明細（Top Nフィルターで件数上限を設ける）

### 4.3 可視化方式の選定方針

- まずPower BI標準ビジュアルで実現できるか確認する
- Waferマップのベクトル表示（矢印プロット）は標準ビジュアルでは困難なため、
  **Deneb（AppSource認定版）+ Vega-Lite** を第一候補とする
- Deneb定義は `deneb/*.json` としてリポジトリ管理し、採否理由は `design.md` に記録する

### 4.4 メジャー要件

- 集計はすべて名前付きメジャーで定義する（計算列・インライン式は使わない）
- 割り算は `DIVIDE` を使用し、複雑な式は `VAR` / `RETURN` で分割する
- 主なメジャー候補:
  - 各成分の平均 / 最大 / 最小 / 標準偏差（Lot・Wafer・号機単位）
  - 残差ベクトル長: `SQRT(x^2 + y^2)` 相当（nm）
  - 選択期間内のWafer数・Lot数・測定点数
- 重複データ（同一WaferId・計測日時・Shot座標・Mark座標）は平均してから
  集計する方針とする（DirectQueryでの実現方法は設計時に確認 → I-006）

#### 共分散楕円の仕様（V-22 / V-23 共通）

X/Y成分のばらつきが「どこで大きいか」と「X/Yが連動して変動しているか」を
1つのビジュアルで読み取るための表示。

- 集計単位:
  - V-22（面内）: Shot位置ごと。同じShot位置の値を選択範囲内のWafer間で集計
  - V-23（Shot内）: Mark位置（Shot中心からの相対座標）ごと。全Shot・全Waferの値を集計
- 各位置で残差Xの分散 `Vx`、残差Yの分散 `Vy`、XとYの共分散 `Cxy` を求める
  （共分散は `E[XY] − E[X]·E[Y]` で計算する）
- 楕円の描画規則:
  - **サイズ**: 共分散行列の固有値の平方根 × 係数k を長軸・短軸にする
    （分散が大きいほど楕円が大きくなる。初期値 k = 2（2σ相当）、design.mdで調整）
  - **傾き**: 主軸方向 `θ = 0.5 × atan2(2·Cxy, Vx − Vy)`（X/Yの相関に応じて傾く）
  - **色**: 相関係数の絶対値 `|r| = |Cxy| / (σx·σy)` をカラースケール（0〜1）に割り当てる
    （無相関 = 薄い色、強い正負相関 = 濃い色。連動して動く場所が一目で分かる）
- Wafer数が3枚未満の位置は統計が不安定なため楕円を表示しない
- 実装方針: 位置ごとの `Vx` / `Vy` / `Cxy` はDAXメジャーで算出し、
  固有値・傾き・色の計算と楕円描画はDeneb（Vega-Lite）側の式で行う
- DirectQueryでの集計負荷が大きいため、必須フィルター（期間・DEVICE_TYPE・
  装置・OPERATION）を絞った状態でのみ描画する

#### 補足事項

- V-16 / V-18 の「補正前の再構成」は、補正量を足すか引くか（符号規約）と
  Mark位置での補正寄与の計算式を装置エンジニアに確認してから実装する（→ I-009）
- V-12 / V-19 など複数Waferをまたぐ集計はDirectQueryの負荷が大きいため、
  期間・Lot数を絞った状態でのみ動作するようフィルター条件を必須にする
- 管理限界線・しきい値（V-06 / V-07）の値は I-005 の確認結果を使う

## 5. データモデル要件

- スタースキーマを基本とする
  - ファクト: Canonビュー、Nikonビュー（それぞれDirectQuery）
  - ディメンション候補: 日付、号機（EQPID）、製品（PRODUCT）、工程（PROCESS）
- リレーションは多対一・単方向とする
- 日付テーブルを必ず用意する。DirectQueryファクトと組み合わせるため、
  日付テーブルはPower Query（M）で作成しDual / Importモードとする
  （複合モデル。計算テーブルは使わない）
- 号機・製品などのディメンションもPower Queryで作成する
  （ソースDBからDISTINCT取得、またはマスタ表があればそれを参照 → I-007）

## 6. 非機能要件

### 6.1 パフォーマンス

- ページロード3秒以内を目標とする
- DirectQueryのため、以下を必須とする:
  - ビジュアルはページあたり8個以下
  - テーブル / マトリックスはTop Nフィルターで表示件数の上限を設ける
  - Waferマップ系ページは必ずWafer 1枚に絞ってから描画する構成にする
  - 期間スライサーに既定値（例: 直近7日）を設定し、全期間クエリを防ぐ
- Deneb含むカスタムビジュアルは本番投入前に表示速度を実測する
- 目標未達の場合は、集計テーブル（Aggregations）やビュー側の事前集計を検討する

### 6.2 セキュリティ

- 接続資格情報・パスワードをリポジトリにコミットしない
- 実データのキャッシュ（`.pbi/cache.abf`）・ローカル設定
  （`.pbi/localSettings.json`）は `.gitignore` に登録する
- 実データのエクスポート（CSV等）はコミットしない。サンプルはダミーデータのみ
- GitHubリポジトリはPrivateとする
- 製造実績データを含むため、RLS（行レベルセキュリティ）の要否を公開前に必ず判断する（→ I-004）

### 6.3 データ更新

- DirectQueryのため、スケジュール更新（データセット更新）は原則不要
- 自動ページ更新（Automatic Page Refresh）を使う場合は、DBへの負荷を考慮して
  間隔を業務側と合意のうえ設定する（→ I-003）

## 7. ソース管理要件

- レポートは**PBIP形式（Power BI Project）**で保存する
- セマンティックモデルはTMDL形式、レポート定義はPBIR形式（JSON）とする
- `.pbix` は配布用成果物とし、リポジトリには含めない
- テーマは `themes/theme.json`、Deneb定義は `deneb/*.json` で管理する
- mainへ直接コミットしない。ブランチ → PR → ユーザー確認 → マージの手順に従う
- PRレビューではTMDL差分（メジャー・リレーション）とレポートJSON差分を確認し、
  見た目の変更はスクリーンショットを添付する

- LithoBooster本体の設定変更・装置へのフィードバック制御

- Zernikeモード分解などの高度な解析計算
  （DirectQuery + DAXでは困難。必要になった場合は別途API / 事前計算の検討とする）
- 社外・インターネットへの公開（社内利用のみを前提とする）
